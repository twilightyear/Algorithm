# [LeetCode] 104. Maximum Depth of Binary Tree


---


### 📌 문제 링크
[LeetCode - Maximum Depth of Binary Tree](https://leetcode.com/problems/maximum-depth-of-binary-tree/description/?envType=problem-list-v2&envId=depth-first-search)


---

### ⚠️ 제약조건
- The number of nodes in the tree is in the range [0, 104].
- -100 <= Node.val <= 100

---


### 🛠️ 풀이 접근 및 핵심 로직
- 본 문제의 풀이 과정은 상당히 단순하다. 루트 노드를 시작으로 DFS 알고리즘을 사용하여, 재귀적으로 접근하며 노드에서 올라올때마다 depth의 최대값을 갱신하면 쉽게 구할 수 있을 것으로 판단했다.
---


### 📊 복잡도 분석
**시간복잡도**
- 재귀적인 탐색을 진행한다. Binary Tree라고 문제에서 명시되어 있으나, 최악의 경우 편향트리의 구조를 가질 수 있으며, 이는 결국 최악의 경우 O(N)의 시간복잡도를 가질 수 있게 한다.


**공간복잡도**
- left와 right등의 임시로 값을 담아두는 것의 공간복잡도는 O(1)이나, 결국 본 문제도 재귀적인 구조로 이루어져있다. 하지만 최악의 경우 호출스택은 O(N)의 공간복잡도를 차지할 수 있다. 결론적으로 O(N)의 공간복잡도를 가진다고 할 수 있겠다. 여기서는 다루지 않았지만 최선의 경우, 완전이진트리 혹은 균형트리의 구조를 가지는 경우에는 O(logN)의 공간복잡도를 가진다)




---


### 📝 회고 / 추후 개선점
- 나의 실행 결과와 다른 사람들의 실행 속도 및 결과를 비교해보니 차이가 다소 존재했다. 일단 나의 코드는 하향식 DFS 구조를 띄는 코드이다. 결국 함수 안에 함수를 선언한다는 점도 오버헤드를 발생시킬 여지가 존재할 수 있겠다. 
- 다른 코드중 인상깊었던 코드는 내가 구현했던것과는 반대방향으로 작동한다. 먼저 재귀의 구조는 같으나, 함수안의 함수를 선언하지는 않았다. 이외에도 leaf 노드에 도달한 후 올라가며 먼저 left=0 과 right=0을  사용하여 max(left,right) 함수로 재귀과정에서 left와 right에 재할당되고, 결국 비교해가면서 올라가는 로직을 가지고 있었다. 두번째 구조가 본 문제 해결 방식에 있어서 훨씬 구조적 안정성과 잠재적 문제를 불러오지 않을 수 있다는 점을 배울 수 있었다.
